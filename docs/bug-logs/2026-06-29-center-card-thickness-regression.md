# 2026-06-29 胶囊粗细与中心卡片回归修复记录

## 问题

1. 胶囊粗细滑块回归：`0%` 时胶囊过细，容易遮住应用图标和中心卡片内容。
2. 音乐应用暂停时，悬浮中心卡片没有显示播放器控件，导致用户无法从中心卡片继续播放、切歌或调整音乐。
3. 顶部灵动岛模式下，中心卡片最大宽度沿用了底部任务栏模式的范围，导致顶部胶囊里显示不完整。

## 根因

1. `CapsuleAppearanceMapper.MapCapsuleHeight` 的最小高度映射被改得过低，`0%` 没有保持为之前用户确认过的安全粗细。
2. `CenterCardPresentationPolicy.Build` 只在 `IsPlaying == true` 时进入音乐详情模式，暂停状态被当成普通应用详情。
3. 中心卡片宽度映射没有区分底部任务栏模式和顶部灵动岛模式，顶部模式也可能使用过大的宽度。

## 修复

1. 胶囊粗细改为安全范围：`0% = 基础高度的 5/6`，`100% = 基础高度`。这样 `0%` 相当于之前视觉上可用的约 `50%` 粗细。
2. 音乐应用只要被识别为音乐应用，悬浮中心卡片就进入播放器详情模式；暂停状态显示 `已暂停`，但仍保留播放、上一首、下一首、音量等控件。
3. 新增 `CenterCardLayoutPolicy`，底部和顶部模式分别计算中心卡片宽度：
   - 底部模式：保持较宽范围，适配任务栏胶囊。
   - 顶部模式：限制到更安全的 `260-320` 宽度区间，避免裁切。

## 验证

1. 针对性测试：胶囊粗细映射、暂停音乐播放器、顶部中心卡片宽度均通过。
2. 完整测试：`102/102` 通过。
3. 构建：`dotnet build` 通过，`0` 警告，`0` 错误。

## 注意

当前本地 PowerShell 截图脚本可以抓取完整物理屏幕尺寸，但对 WPF 半透明分层窗口的捕获不稳定。视觉自检时需要结合窗口坐标枚举、用户截图或后续更可靠的截图工具判断，不要单独依赖该脚本判断胶囊是否显示。

## 追加修复

用户反馈短胶囊和顶部模式下中心卡片仍然显示不完整后，重新对比 `origin/win-ui1.0-优化版`：

1. 胶囊粗细映射恢复为 GitHub 最新版逻辑：`0% = 53.3/80`，`50% = 66.7/80`，`100% = 80/80`，滑块变化更明显。
2. 中心卡片宽度改为受胶囊剩余空间约束：底部模式缩到最短时，中心卡片最大宽度不再超过 `320`，避免压住右侧系统区。
3. 顶部灵动岛模式主栏应用位从 `5` 个收紧到 `3` 个，多余应用交给收纳夹，给中心卡片和系统区留出稳定空间。
4. 音乐播放器按钮增加自适应密度：宽卡片显示完整按钮，窄卡片隐藏播放顺序/音量，极窄时至少保留播放暂停按钮。

## 再次回归修复

用户继续反馈：粗细滑块拖动后不是实时变化，短胶囊和顶部胶囊的中心卡片仍然显示异常。再次排查后确认：

1. 粗细滑块已经更新配置，但 `CapsuleBorder.Height` 可能仍被之前的高度动画控制，所以新高度会等动画结束或后续刷新后才体现。
2. 主胶囊内部使用 `DockPanel LastChildFill=True`，中心卡片作为最后一个子元素会被 DockPanel 强制填充剩余空间，导致我们计算出的中心卡片宽度没有稳定生效。

修复：

1. `ApplyCapsuleThickness` 在设置高度前先清除高度动画：`CapsuleBorder.BeginAnimation(HeightProperty, null)`，保证滑块拖动时立即反映到 UI。
2. 外观滑块改成先刷新 UI、再保存配置，减少磁盘写入对视觉反馈的影响。
3. 主胶囊内部布局从 `DockPanel LastChildFill=True` 改为四列 `Grid`：左应用区 / 分隔线 / 中心卡片 / 右系统区。中心卡片固定在中间列并居中，不再被 DockPanel 撑开。

## 中心卡片比例缩放修复

用户继续反馈：最短长度时中心卡片两端仍然会被左右区域覆盖。原来的修复仍然在使用“最小宽度/最大宽度区间插值”，这会导致胶囊缩短时中心卡片没有严格按胶囊长度同步缩小。

修复：

1. `CenterCardWidthPercent` 改为表示中心卡片和胶囊长度之间的比例档位。
2. 中心卡片宽度计算改为：`中心卡片宽度 = 当前胶囊宽度 * 当前比例`。
3. 当前比例范围为 `18% - 40%`，默认 `58%` 对应约 `30.8%` 胶囊长度。
4. 因此同一个中心卡片设置下：
   - `1920` 宽胶囊时中心卡片约 `590.6`。
   - `760` 宽短胶囊/顶部胶囊时中心卡片约 `233.8`。

验证：

1. 针对性测试：`10/10` 通过。
2. 完整测试：`106/106` 通过。
3. 构建：`dotnet build` 通过，`0` 警告，`0` 错误。

## 中心卡片可用槽位二次修复

用户继续反馈：`中心卡片长度最大 + 胶囊长度最小` 时，中心卡片两端仍然会被左右应用区/系统区覆盖。

根因：

1. 上一版虽然把中心卡片宽度限制到中间列可用宽度内，但没有把 `ActiveAppSummaryPanel` 自身 `Margin="18,0"` 计算进去。
2. WPF 布局中元素的 `Width` 和 `Margin` 会一起占用父容器空间，所以 `Width = 槽位宽度 - 16` 仍然会因为左右共 `36px` 的外边距而被裁切。

修复：

1. 中心卡片最终宽度改为：`min(胶囊比例宽度, 中间槽位宽度 - 中心卡片左右外边距 36 - 安全间距 16)`。
2. 新增测试覆盖 `760px` 短胶囊、`250px` 中间槽位、`100%` 中心卡片长度的极限场景，确保宽度会压缩到 `198px`，连同左右外边距后仍不会越界。

验证：

1. 针对性测试：`14/14` 通过。
2. 完整测试：`109/109` 通过。
3. 构建：`dotnet build` 通过，`0` 警告，`0` 错误。

## 播放音乐未悬浮时歌词不显示

用户反馈：网易云音乐正在播放时，鼠标没有悬浮中心卡片，中心卡片只显示应用图标和两侧音符，没有显示歌词。

根因：

1. `CenterCardPresentationPolicy` 仍然会进入 `MusicLyricsMarquee`，策略层没有忘记“未悬浮显示歌词”的需求。
2. 真正问题在 WPF 布局：短中心卡片下，`CenterCardLyricsLayer` 的左右外边距、应用图标、左右音符装饰一起占用了大部分宽度，歌词文本视口被压到几乎不可见。
3. 跑马灯动画从卡片右侧外部开始进入，短文本也会有一段空窗期，看起来像没有歌词。

修复：

1. 新增 `CenterCardLayoutPolicy.GetLyricsLayout`，按中心卡片宽度决定歌词层密度：
   - `Full`：保留左右音符。
   - `Compact`：保留左侧音符，隐藏右侧音符。
   - `Minimal`：隐藏两侧音符，把空间让给歌词。
2. `MainWindow` 在中心卡片宽度变化和展示状态刷新时调用 `ApplyCenterCardLyricsLayout`。
3. 歌词跑马灯改为：文本能放下时直接显示；只有文本超出视口时才从当前位置向左滚动。
4. 给歌词视口增加 `CenterCardLyricsViewport` 命名，方便测量和视觉契约测试。

验证：

1. 红灯测试：新增 `GetLyricsLayout_PrioritizesTextSpaceWhenCenterCardIsNarrow` 后，先失败于 `CenterCardLayoutPolicy` 缺少 `GetLyricsLayout`。
2. 针对性测试：中心卡片布局、展示策略、视觉契约共 `26/26` 通过。
3. 完整测试：`112/112` 通过。
4. 构建：`dotnet build` 通过，`0` 警告，`0` 错误。
