# DynamicIslandBar 可扩展胶囊壳层与运行中应用设计说明（中文）

> 状态：已确认的设计说明，作为后续实现与演进的主规范  
> 日期：2026-06-25  
> 项目：`DynamicIslandBar`  
> 说明：本设计覆盖并替代前一版仅聚焦“运行中应用 + 双形态”的实现范围，新增“最终版本导向的代码框架”要求。

## 1. 设计目标

本轮不是单纯给当前原型继续堆功能，而是要把 `DynamicIslandBar` 的代码框架升级为“可持续演进的最终版起点”。

当前设计目标分为两层：

1. `框架目标`
   把当前项目从“单窗口 + 直接写逻辑”的原型结构，升级为一个可扩展的胶囊壳层框架。

2. `本期功能目标`
   在这个可扩展框架上，先落地一批真实可见的用户能力：
   - 顶部 / 底部双形态
   - 实时运行中应用图标
   - 应用隐藏 / 显示 / 打开 / 关闭 / 喜好
   - 主栏溢出收纳文件夹
   - 方格按钮升级为应用管理面板
   - 最小可用主题预设切换

最终要求是：当前写进去的代码，后续可以平滑升级到以下最终版本目标，而不需要推翻重做：
- 任意位置拖动停放
- 拖动边缘自由缩放
- 用户上传背景图片
- 完整设置页
- 轻量安装包与发布优化
- 启动速度与内存专项优化

## 2. 本轮范围定义

### 2.1 本轮必须落地

- 胶囊框架分层
- 顶部 / 底部双形态
- 拖动切换顶部 / 底部吸附
- 运行中应用发现与图标显示
- 单击激活 / 再次单击最小化
- 右键菜单：隐藏图标 / 显示到胶囊、关闭应用 / 打开应用、添加到喜好 / 取消喜好
- 主栏溢出收纳文件夹
- 方格按钮应用管理面板
- 2-3 个内置风格预设
- 主题切换临时入口
- 统一配置持久化

### 2.2 本轮明确不做

- 任意位置自由停放
- 任意边缘拖拽缩放
- 用户上传背景图片
- 正式设置页
- 多显示器支持
- 自动隐藏系统任务栏
- 安装包 / 卸载器 / 签名 / 自动更新
- 启动速度专项优化
- 内存专项优化

### 2.3 本轮虽然不做，但必须预留接口

- 自由拖动后的持久化位置
- 自定义尺寸与边缘缩放
- 背景图片路径与渲染能力
- 设置页对布局与主题配置的统一读写
- 发布版可裁剪、可配置化的启动与资源策略

## 3. 产品行为目标

### 3.1 胶囊的两种正式形态

当前版本只支持两个正式模式：

1. `BottomTaskbar`
   - 胶囊默认位于主屏幕底部
   - 宽度明显长于当前版本
   - 视觉上覆盖在系统任务栏区域之上
   - 作为“自定义任务栏”使用

2. `TopIsland`
   - 胶囊吸附到主屏幕顶部居中
   - 宽度自动收紧，接近灵动岛形态
   - 保留系统区与运行中应用区

当前版本只允许通过拖动在这两个停靠位之间切换，不保留中间自由停靠状态。

### 3.2 系统区在两种模式下都要保留

无论顶部还是底部模式，都需要继续保留：
- WiFi
- 音量
- 电池
- 时间
- 方格按钮

区别只在布局密度和弹层方向，不在功能范围。

### 3.3 运行中应用区

胶囊左侧将从“写死的演示图标”升级为“动态运行中应用区”。

自动发现范围：
- 只取有可见顶层主窗口的应用
- 过滤系统壳窗口
- 过滤工具窗口与隐藏窗口
- 过滤 `DynamicIslandBar` 自身

行为规则：
- 一个应用只显示一个图标
- 单击：不在前台 / 已最小化时恢复并激活
- 再次单击：如果代表窗口已在前台，则最小化
- 右键：弹出状态化菜单

### 3.4 应用右键菜单

每个应用图标都必须支持以下动作：

- `隐藏图标` / `显示到胶囊`
- `关闭应用` / `打开应用`
- `添加到喜好` / `取消喜好`

行为解释：
- `隐藏图标`：只从主栏移除，应用继续运行
- `显示到胶囊`：将隐藏应用恢复到主栏
- `关闭应用`：关闭当前应用
- `打开应用`：根据已知路径重新启动
- `添加到喜好`：固定为喜好应用
- `取消喜好`：取消固定

### 3.5 喜好应用

喜好应用是未来“固定到胶囊”的基础能力。

规则：
- 喜好应用即使未运行，也可以保留为条目
- 未运行喜好应用显示为弱化态
- 单击未运行喜好应用时，可直接启动
- 如果喜好应用被关闭，不应彻底消失，而应转成“未运行但可再次打开”的状态

### 3.6 主栏溢出收纳文件夹

主栏不应该无限扩张挤压系统区。

改为：
- 主栏只显示当前模式允许的有限数量应用
- 一旦超过容量，最后一个位置改为 `收纳文件夹`
- 鼠标悬浮收纳图标时，弹出收纳面板
- 收纳面板显示所有超出主栏容量的应用
- 收纳面板中的应用支持与主栏完全一致的点击与右键行为

### 3.7 方格按钮升级

方格按钮保留，但其悬浮面板升级为完整应用管理面板。

至少包含三个分区：
- `运行中应用`
- `已隐藏应用`
- `喜好应用`

规则：
- 悬浮按钮时显示面板
- 鼠标移入面板后可继续交互
- 鼠标移出按钮和面板后自动关闭
- 单击方格按钮时，仍保留当前“打开任务管理器”的行为

### 3.8 弹层定位规则

底部模式：
- WiFi / 音量 / 应用管理 / 收纳文件夹面板优先向上弹出

顶部模式：
- 所有面板优先向下弹出或自动调整位置
- 任何情况下都不能顶出屏幕

## 4. 代码框架目标

### 4.1 总原则

本轮不做一次性大重构，但必须把当前代码从“功能直接堆到 `MainWindow`”的状态，拆成可持续扩展的壳层结构。

原则如下：
- `MainWindow` 只做宿主与交互编排
- 布局逻辑单独管理
- 主题逻辑单独管理
- 运行中应用逻辑单独管理
- 配置持久化单独管理
- 底层 Windows 窗口操作继续隔离在底层服务中

### 4.2 MainWindow 的职责

`MainWindow` 负责：
- 承载胶囊窗口
- 接收鼠标悬浮 / 点击 / 拖动
- 管理 Popup 的开关
- 将服务层数据绑定或渲染到界面
- 响应布局切换与主题切换

`MainWindow` 不再直接承担：
- 运行中应用归并逻辑
- 配置结构拼装逻辑
- 主题预设定义
- 顶层布局模式计算

## 5. 布局层设计

### 5.1 新增 `CapsuleLayoutManager`

新增一个专门的布局管理层，负责“胶囊当前应该长什么样、摆在哪里、各区域容量是多少”。

当前只支持两个模式：
- `BottomTaskbar`
- `TopIsland`

### 5.2 必须抽象的布局概念

当前实现就要引入以下结构概念：

- `CapsuleMode`
  - 当前值：`BottomTaskbar`、`TopIsland`
  - 后续可扩展：`Floating`

- `CapsulePlacement`
  - 当前保存吸附位与窗口 bounds
  - 后续可扩展为任意拖动后的自由位置

- `CapsuleSizeProfile`
  - 当前区分底部长条模式与顶部紧凑模式
  - 后续可扩展为用户自定义尺寸

- `LayoutMetrics`
  - 集中定义宽高、图标大小、边距、系统区间隔、主栏可见应用容量、收纳阈值、Popup 偏移量

### 5.3 当前布局层必须解决的问题

- `MainWindow.xaml` 中大量固定宽度、固定间距、固定弹层偏移值，需要改成由 `LayoutMetrics` 驱动
- 顶部 / 底部两种模式的宽高和容量必须由布局层统一计算
- 当前版本虽然不做任意缩放，但布局层必须支持“未来尺寸变化后重新计算 UI”

## 6. 主题层设计

### 6.1 新增 `CapsuleThemeManager`

新增一个主题层，负责将“风格预设”映射为胶囊实际使用的视觉变量。

### 6.2 本轮主题范围

本轮至少支持 2-3 个内置风格预设，例如：
- `ClassicDark`
- `GlassGreen`
- `SoftLight`

主题层至少统一管理：
- 胶囊背景色
- 透明度
- 阴影强度
- 圆角
- 高亮色
- 面板背景色
- 边框色
- 图标激活色

### 6.3 背景图字段预留

虽然本轮不做背景图上传，但主题配置中必须预留：
- `BackgroundImagePath`
- `BackgroundImageOpacity`
- `BackgroundImageStretchMode`

本轮只要求：
- 配置结构具备这些字段
- 应用主题时能够安全忽略未启用的背景图设置

### 6.4 主题切换入口

本轮不做正式设置页。

采用最小可用入口：
- 在胶囊右键菜单中增加 `风格` 子菜单
- 可直接切换内置预设

这样可以验证主题系统不是空架子，同时不提前引入完整设置页复杂度。

## 7. 运行中应用层设计

### 7.1 新增 `RunningAppsService`

运行中应用相关逻辑不继续堆在 `MainWindow` 和 `WindowManager` 中。

新增一个高层服务，负责：
- 枚举窗口
- 读取进程信息
- 解析可执行路径
- 提取应用图标
- 按应用归并
- 组合隐藏 / 喜好 / 启动路径状态
- 产出主栏应用、收纳应用、管理面板应用快照

### 7.2 `WindowManager` 继续做底层

现有 `WindowManager` 保留为底层系统交互层，可扩展能力包括：
- 枚举窗口及更多元数据
- 判断前台窗口
- 恢复窗口
- 激活窗口
- 最小化窗口

但不再承担：
- 喜好逻辑
- 主栏 / 收纳 / 管理面板划分逻辑
- 主题与布局逻辑

### 7.3 应用数据模型

建议新增应用级模型，至少包含：

```text
AppEntry
- AppId
- DisplayName
- ExePath
- IconSource
- IsRunning
- IsFavorite
- IsHiddenInCapsule
- WindowHandles
- RepresentativeWindowHandle
```

说明：
- `AppId` 优先使用标准化 exe 路径
- 路径取不到时退化为进程名
- `RepresentativeWindowHandle` 用于“激活 / 最小化”切换

### 7.4 多窗口策略

本轮先采用简单稳定策略：
- 一个应用即使有多个窗口，也只显示一个图标
- 只选一个代表窗口
- 不做窗口组预览，不做每窗口切换

### 7.5 刷新策略

先采用低频轮询：
- 约 `800ms` 到 `1200ms` 一次

原因：
- 简单
- 易稳定
- 足够满足当前交互

后续如果要进一步优化实时性，再增加 WinEventHook 等事件驱动机制。

## 8. 配置层设计

### 8.1 新增统一配置层

新增一层统一的胶囊配置服务，用于保存和读取以下状态：

- 胶囊模式
- 当前风格预设
- 背景图字段（预留）
- 喜好应用
- 已隐藏应用
- 已知启动路径

### 8.2 与权限配置分离

现有 `PermissionService` 保持独立，不与通用胶囊配置混在一起。

理由：
- 权限是系统访问许可
- 胶囊配置是用户自定义状态

两者生命周期和职责不同，应避免未来膨胀成一个不可维护的配置文件。

## 9. 未来版本预留点

### 9.1 为任意拖动预留

虽然本轮只允许顶部 / 底部吸附，但布局层和配置层必须预留：
- 自由位置字段
- 拖动结束后的停放策略入口
- 非吸附模式的 bounds 存储入口

### 9.2 为任意缩放预留

虽然本轮不实现边缘拖拽缩放，但布局层必须预留：
- 用户尺寸模型
- 图标区 / 系统区在不同尺寸下的容量重算逻辑
- 最小尺寸与最大尺寸概念

### 9.3 为背景图与设置页预留

主题层与配置层必须能承接：
- 用户上传图片路径
- 透明度与裁剪策略
- 设置页对主题配置的实时预览和保存

## 10. 文件职责建议

本轮建议逐步形成如下职责边界：

- `MainWindow.xaml`
  - 胶囊基础结构
  - Popup 宿主
  - 动态容器

- `MainWindow.xaml.cs`
  - 交互编排
  - 事件转发
  - 调用布局层、主题层、运行中应用层、配置层

- `CapsuleLayoutManager`
  - 模式、尺寸、位置、容量计算

- `CapsuleThemeManager`
  - 风格预设定义
  - 当前主题值生成与应用

- `RunningAppsService`
  - 应用发现、状态合并、图标与操作入口

- `CapsuleConfigService`
  - 胶囊配置的统一读写

- `WindowManager`
  - Windows 窗口底层操作

## 12. 本轮成功标准

本轮完成后，项目应该达到以下状态：

- 从代码结构上，不再是一次性原型
- 后续能自然接入任意拖动、任意缩放、背景图、自定义设置页
- 当前版本用户已经可以看到真实可用的：
  - 双形态胶囊
  - 运行中应用主栏
  - 收纳文件夹
  - 管理面板
  - 喜好与隐藏
  - 主题预设切换

换句话说：

`这轮交付的是“最终版本的框架起点 + 一批真实功能”，而不是“最终版本全部能力”。`

## 13. 推荐实现顺序

建议后续实现顺序如下：

1. 先抽配置层与布局层基础模型
2. 再抽主题层并接入最小预设切换
3. 再实现运行中应用服务与主栏图标
4. 再实现收纳文件夹与方格管理面板
5. 再接顶 / 底双模式切换与吸附
6. 最后做集成回归与测试补齐

这样可以在不打断现有可运行状态的前提下，逐步替换掉原型代码结构。
