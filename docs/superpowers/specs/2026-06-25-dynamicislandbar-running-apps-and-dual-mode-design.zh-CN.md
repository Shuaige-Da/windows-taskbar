# DynamicIslandBar 运行中应用与双形态设计说明（中文）

> 状态：已确认的设计说明，下一步可据此编写实现计划  
> 日期：2026-06-25  
> 项目：`DynamicIslandBar`

## 1. 目标

为 `DynamicIslandBar` 增加一套可交付的“运行中应用”能力，让胶囊既能像一个轻量级 Windows 自定义任务栏，又能在顶部切换成紧凑版“灵动岛”。

本设计包含：
- 胶囊内实时显示运行中应用图标
- 应用溢出收纳
- 应用隐藏 / 显示 / 喜好 / 打开 / 关闭交互
- 应用管理面板
- 胶囊双形态：底部任务栏模式、顶部灵动岛模式

本设计暂不包含：
- 完整设置界面
- 拖拽排序
- 多显示器支持
- Windows 10 正式发布支持
- 多窗口预览等复杂窗口组能力

## 2. 产品意图

胶囊不再只是左侧 3 个写死图标的演示条。

它需要变成一个真实的桌面壳层，并且先支持两种稳定形态：

1. `底部任务栏模式`
   默认位于桌面底部，视觉上覆盖系统任务栏区域，承担“自定义任务栏”的角色。

2. `顶部灵动岛模式`
   当用户把胶囊拖到顶部时，自动吸附并切换为更紧凑的灵动岛形态。

两种模式共用同一套应用状态、交互规则和系统功能。

## 3. 用户可见行为

### 3.1 自动发现哪些应用

胶囊只自动发现“有可见顶层主窗口”的桌面应用，尽量接近 Windows 任务栏的感受。

典型例子：
- 微信
- QQ
- 抖音
- Codex
- 浏览器
- 编辑器

默认排除：
- 系统壳窗口
- 隐藏窗口 / 工具窗口
- 当前应用自身：`DynamicIslandBar`
- 没有可见主窗口的后台进程

### 3.2 胶囊主栏中的应用图标

胶囊左侧改成动态应用区。

行为规则：
- 实时显示运行中应用图标
- 一个应用只显示一个图标，不按窗口条目逐个显示
- 单击应用图标时执行窗口状态切换：
  - 如果窗口当前最小化，或不在前台，则恢复并激活
  - 如果代表窗口当前已经在前台，则最小化
- 右键应用图标时弹出菜单

### 3.3 应用右键菜单

每个应用图标都支持状态化右键菜单。

必需菜单项：
- `隐藏图标` / `显示到胶囊`
- `关闭应用` / `打开应用`
- `添加到喜好` / `取消喜好`

语义定义：
- `隐藏图标`：只是不在胶囊主栏显示，应用继续运行
- `显示到胶囊`：把已隐藏应用重新放回胶囊主栏
- `关闭应用`：关闭该应用进程
- `打开应用`：根据已知可执行路径重新启动应用
- `添加到喜好`：把应用固定为喜好应用
- `取消喜好`：取消固定

菜单项可用性：
- `打开应用` 只有在可执行路径已知时才可用
- `关闭应用` 只对正在运行的应用显示
- 非运行中的喜好应用，如果有启动路径，也显示 `打开应用`

### 3.4 喜好应用

喜好应用就是后续“固定到胶囊”的基础能力。

行为规则：
- 喜好应用即使当前未运行，也可以继续保留
- 未运行的喜好应用显示为弱化态
- 单击未运行的喜好应用时，直接启动应用
- 如果一个喜好应用被关闭，它不会从体系里消失，而是转为“未运行但可再次启动”

### 3.5 已隐藏应用

隐藏状态只影响“是否显示在胶囊主栏”。

行为规则：
- 隐藏不会终止应用
- 已隐藏应用仍然可以在其他管理面板中被看到和恢复
- 已隐藏的喜好应用仍然是喜好应用，只是不显示在主栏

### 3.6 主栏溢出收纳文件夹

胶囊主栏不能无限变长直到挤压时间区和系统区。

改成如下规则：
- 胶囊主栏只显示当前布局允许的有限数量应用图标
- 一旦超过上限，最后一个位置变成 `应用收纳文件夹` 图标
- 鼠标悬浮在收纳文件夹上时，在胶囊上方弹出悬浮面板
- 面板中显示所有超出主栏容量的应用
- 收纳面板中的应用和主栏应用拥有完全一致的单击、右键能力

交互规则：
- 悬浮到收纳图标 -> 打开收纳面板
- 鼠标移入收纳面板 -> 保持可交互
- 鼠标移出图标和面板 -> 自动隐藏

### 3.7 方格按钮的应用管理面板

现在的方格按钮保留，但职责升级。

它不再只是一个可见窗口的纯文本列表，而是完整的应用管理视图。

分区：
- `运行中应用`
- `已隐藏应用`
- `喜好应用`

面板规则：
- 像 WiFi / 声音面板一样，从胶囊上方悬浮出现
- 不与胶囊本体重叠
- 支持悬浮打开、悬浮进入后继续交互
- 使用“图标 + 名称”的应用项，而不是只显示文本
- 从方格按钮悬浮打开
- 方格按钮单击仍保留当前行为：打开任务管理器

收纳文件夹与方格按钮的分工：
- `收纳文件夹`：访问当前主栏装不下的运行中应用
- `方格按钮`：查看完整状态和执行统一管理

## 4. 胶囊双形态

### 4.1 底部任务栏模式

底部模式为默认启动模式。

行为规则：
- 胶囊位于主屏幕底部
- 胶囊宽度根据桌面任务栏视觉宽度自适应，明显长于现在
- 胶囊视觉上覆盖在系统任务栏区域之上
- 本次功能不要求自动隐藏系统任务栏
- 胶囊在这个模式下承担“自定义任务栏”的角色

这个模式优先承载：
- 运行中应用区
- 溢出收纳文件夹
- WiFi / 音量 / 电池 / 时间 / 方格按钮

### 4.2 顶部灵动岛模式

当用户把胶囊拖到顶部吸附区后，切换为顶部模式。

行为规则：
- 胶囊吸附到主屏幕顶部中间
- 胶囊宽度自动变小，成为紧凑版灵动岛
- 以下系统区能力全部保留在胶囊中：
  - WiFi
  - 音量
  - 电池
  - 时间
  - 方格按钮
- 运行中应用能力保留
- 溢出收纳能力保留
- 所有弹层必须重新定位，避免从屏幕顶部溢出；需要时改为向下弹出

顶部模式改变的是布局密度，不是功能范围。

### 4.3 拖动与吸附规则

V1 只支持两个最终停靠点：
- 顶部
- 底部

规则：
- 用户可以直接拖动胶囊
- 松手位置接近顶部阈值时，吸附到顶部模式
- 松手位置接近底部阈值时，吸附到底部模式
- 中间区域不保留自由悬浮状态

这样状态机简单，V1 更稳定。

## 5. 系统结构

实现上需要把“窗口发现、应用状态、应用控制、UI 布局”拆开。

### 5.1 新服务：`RunningAppsService`

新增一个专门服务，负责发现并组装“运行中应用”。

职责：
- 枚举可见顶层窗口
- 把窗口归并成应用级条目
- 读取进程信息
- 尽可能读取可执行路径
- 提取应用图标
- 给 UI 输出可刷新快照

这个服务不直接持有 WPF 控件。

### 5.2 现有 `WindowManager` 的边界

`WindowManager` 继续作为底层 Windows 交互层。

它可以扩展负责：
- 枚举窗口并返回更多元数据
- 判断前台窗口
- 恢复并激活窗口
- 最小化窗口

但它不应该承载：
- 喜好 / 隐藏状态逻辑
- WPF 菜单逻辑

### 5.3 应用状态层

增加一层轻量状态管理，保存：
- Hidden app ids
- Favorite app ids
- Known launch paths
- Capsule mode：top 或 bottom

启动时加载，运行中更新，退出前持久化。

### 5.4 UI 组合层

`MainWindow` 主要负责：
- 渲染应用图标
- 渲染收纳文件夹
- 管理悬浮弹层
- 切换顶部 / 底部布局
- 生成应用右键菜单

尽量把业务逻辑放到服务层，方便测试。

## 6. 数据模型

建议新增应用级模型，类似：

```text
AppEntry
- AppId
- DisplayName
- ExePath
- IconSource
- IsRunning
- IsFavorite
- IsHiddenInCapsule
- WindowHandles
- RepresentativeWindowHandle
```

字段含义：
- `AppId`：稳定身份标识，优先用归一化后的 exe 路径，取不到时退化到进程名
- `DisplayName`：给用户看的应用名称
- `ExePath`：用于“打开应用”
- `IconSource`：给 WPF 渲染的图标源
- `IsRunning`：当前是否正在运行
- `IsFavorite`：是否为喜好应用
- `IsHiddenInCapsule`：是否在胶囊主栏隐藏
- `WindowHandles`：该应用对应的可见窗口句柄集合
- `RepresentativeWindowHandle`：用于激活 / 最小化切换的代表窗口

## 7. 发现与刷新规则

### 7.1 发现规则

发现规则如下：
- 只取可见顶层窗口
- 标题不能为空
- 过滤系统壳窗口
- 过滤本应用自己
- 按应用归并，而不是按窗口逐条显示

V1 的多窗口处理规则：
- 一个应用即便有多个窗口，也只形成一个 `AppEntry`
- 选一个代表窗口用于激活 / 最小化
- V1 不做窗口级预览或窗口列表切换

### 7.2 刷新策略

V1 先采用低频轮询。

目标区间：
- 大约 `800ms` 到 `1200ms` 一次

原因：
- 结构简单
- 足够稳定
- 响应对 V1 来说已足够
- 避免过早引入复杂事件钩子体系

后续再考虑事件驱动增强。

### 7.3 失败容错

如果某个应用在读取以下信息时失败：
- 进程信息
- 路径
- 图标
- 窗口句柄

则只影响这个应用条目，不允许拖垮整次刷新。

## 8. 持久化

本功能使用轻量 JSON 配置文件持久化。

必须保存的字段：
- `FavoriteApps`
- `HiddenAppIds`
- `KnownLaunchPaths`
- `CapsuleMode`

后续可扩展：
- `DisplayOrder`

规则：
- 一旦发现合法可执行路径，就更新 `KnownLaunchPaths`
- 隐藏和喜好状态都属于用户状态，不由系统推断
- 模式状态在下次启动时恢复

V1 暂不做手工编辑这些状态的专门设置界面。

## 9. UI 布局细节

### 9.1 替换现在的演示 Dock

当前左侧 3 个写死的示例卡片，替换为动态应用容器。

V1 视觉规则：
- 统一的圆角图标按钮
- 顶部和底部模式都保持一致的风格体系
- 未运行的喜好应用使用低透明度或弱化态
- 已隐藏应用不渲染到胶囊主栏

### 9.2 收纳文件夹面板

新增一个收纳文件夹 Popup，交互规则和现在的 WiFi / 声音 / 方格面板一致。

要求：
- 悬浮图标打开
- 面板可继续交互
- 移出后自动关闭
- 不与胶囊重叠
- 顶部模式下必须向下弹出或做等效重定位，确保不越界

### 9.3 方格按钮管理面板布局

方格按钮面板从“纯文本窗口列表”升级为“应用管理面板”。

每个条目至少显示：
- 图标
- 名称
- 必要状态提示

需要支持：
- 单击应用行为
- 右键菜单行为
- 运行 / 隐藏 / 喜好状态区分
- 对已隐藏应用提供 `显示到胶囊` 的恢复入口

### 9.4 模式差异

底部模式：
- 更长的宽度
- 更高的主栏可见应用容量
- 更接近任务栏节奏

顶部模式：
- 更短的宽度
- 更低的主栏可见应用容量
- 更紧凑的间距

两种模式都必须保留：
- WiFi / 音量 / 电池 / 时间 / 方格按钮
- 主栏应用区
- 溢出收纳能力

## 10. 交互语义

### 10.1 单击

对于运行中的应用：
- 如果代表窗口不在前台或处于最小化，则恢复并激活
- 如果代表窗口已在前台，则最小化

对于未运行的喜好应用：
- 如果已知启动路径，则直接启动

### 10.2 右键

右键弹出 WPF ContextMenu。

菜单文案根据状态切换：
- 隐藏 / 显示
- 运行 / 未运行
- 喜好 / 非喜好

### 10.3 关闭 / 打开后的行为

如果一个非喜好应用被关闭：
- 下次刷新发现它已停止后，就从主栏移除

如果一个喜好应用被关闭：
- 仍保留为“未运行的喜好应用”
- 可以再次打开

## 11. 边界情况

### 11.1 取不到可执行路径

如果拿不到 exe 路径：
- 运行中时仍然可以显示
- 如果有有效窗口，仍可支持激活 / 最小化
- `打开应用` 需要禁用
- `添加到喜好` 也可以禁用，以避免后续无法重新打开

### 11.2 取不到图标

图标降级顺序：
1. 应用专属图标
2. exe 默认图标
3. 通用占位图标

### 11.3 过滤自身

`DynamicIslandBar` 不能出现在：
- 主栏应用区
- 收纳文件夹
- 方格管理面板

### 11.4 应用太多

如果运行中应用数量超过主栏容量：
- 主栏只显示上限减一的数量
- 最后一个位置固定给收纳文件夹

### 11.5 喜好 + 隐藏

如果一个应用同时是“喜好”又是“隐藏”：
- 它不显示在胶囊主栏
- 但仍出现在管理面板中
- 如果当前未运行且有启动路径，仍可重新打开

## 12. 错误处理

这个功能不能让现有胶囊变得不稳定。

要求：
- 应用发现过程不能阻塞 UI 线程
- 悬浮面板刷新要保持流畅
- 图标提取失败不能导致界面崩溃
- 关闭应用失败时要优雅降级
- 打开应用失败时要优雅降级

V1 以静默降级和后续日志为主，不做侵入式弹窗。

## 13. 测试策略

### 13.1 逻辑测试

补充测试：
- 多窗口归并为应用
- 将持久化的隐藏 / 喜好状态合并进实时应用快照
- 收纳分流逻辑
- 喜好应用关闭后保留逻辑
- 右键菜单状态映射逻辑

### 13.2 UI / 集成验证

验证内容：
- 底部模式布局正常
- 顶部模式布局正常
- 拖到顶部可以正确吸附
- 拖到底部可以正确吸附
- 收纳面板悬浮行为正常
- 方格管理面板正常
- 不影响现有 WiFi / 音量 / 电池交互
- 单击应用图标的激活 / 最小化切换正常

## 14. 本轮明确不做

本轮不包含：
- 拖拽排序
- 多窗口分组预览
- 多显示器支持
- 自动隐藏系统任务栏
- 直接镜像 Windows 任务栏内部按钮
- 本功能专属正式设置界面
- 复杂拖动过渡动画

## 15. 推荐实施顺序

建议后续实现计划按下面的顺序拆：

1. 补底层窗口 / 应用发现能力
2. 加应用状态持久化
3. 加动态应用图标渲染
4. 加收纳文件夹交互
5. 升级方格按钮为完整应用管理面板
6. 加顶部 / 底部双模式布局与拖动吸附
7. 补测试与回归验证

这样能尽量利用现有代码结构，避免一次性大改导致不稳定。
